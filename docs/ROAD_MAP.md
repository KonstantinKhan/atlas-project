# 🗺️ Дорожная карта развития системы PPM

_(Project & Portfolio Management)_

Документ ориентирован на:
- постепенное наращивание сложности
- сохранение чистой архитектуры
- поддержку как UI, так и доменной модели
---
## 🧱 ЭТАП 0. Архитектурный фундамент
### Цель

Создать **устойчивую основу**, на которую можно наращивать функциональность без рефакторинга «через боль».
### Ключевые решения

- разделение:
    - Task ↔ Schedule
    - UX-команды ↔ доменные команды
    - Calendar ↔ ProjectPlan
- план = агрегат, живущий **в рамках запроса**
- календарь = **внешний, кэшируемый контекст**
### Что должно быть готово

- доменные модели:
    - `ProjectTask`
    - `TaskSchedule`
    - `ProjectPlan`
    - `TaskDependency`
    - `TimelineCalendar` _(в дорожной карте — `WorkCalendar`, в коде — `TimelineCalendar`)_
- UX-команды:
    - Create / Rename / ChangeStartDate / ChangeEndDate / Move / Resize
    - _Примечание: исходная команда «Schedule» декомпозирована на `ChangeStartDate` и `ChangeEndDate` — так реализовано в коде_
- чёткий pipeline:
    ```
    UI → UX Command → API → Domain → DB → DTO → UI
    ```

📌 **Без этого дальше идти нельзя**

---
## 🧩 ЭТАП 1. Планирование одного проекта (Core MVP)

### Цель

Полноценное **интерактивное планирование одного проекта** без ресурсов.

---
### 1.1 Task Pool (список задач)

**Функции**
- создание задач (title обязателен)
- переименование
- удаление
- задачи могут существовать без дат

**Почему важно**
- разделяет _что делать_ и _когда делать_
- снижает когнитивную нагрузку
---
### 1.2 Gantt-планирование

**Функции**
- назначение задачи на таймлайн
- drag / resize
- планирование вперёд
- планирование назад _(от конечной даты к начальной: задаётся end → вычисляется start)_
- реакция на нерабочие дни
**Инварианты**
- любые изменения → пересчёт через календарь
- UI не считает даты
---
### 1.3 Зависимости

**Функции**
- FS (Finish–Start) как минимум
- визуальные связи
- пересчёт цепочек
- смена типа зависимости (FS/SS/FF/SF) через UI

📌 На этом этапе **без критического пути**

#### Backlog

| Задача | Описание | Статус |
|--------|----------|--------|
| Смена типа зависимости | Попап при клике по стрелке: выбор типа (FS/SS/FF/SF). Пересчёт лага и каскад после смены. Backend: endpoint PATCH /dependencies, Domain: метод changeDependencyType в ProjectPlan. Frontend: расширить DependencyActionPopover селектором типа. | Done |

---
### Результат этапа

> У тебя система уровня **MS Project Lite** для одного проекта

---
## 📐 ЭТАП 2. Графы и алгоритмы планирования

### Цель

Сделать план **интеллектуальным**, а не просто визуальным.

---

### 2.1 Граф задач

**Что появляется**
- DAG задач
- топологическая сортировка
- валидация циклов
---
### 2.2 Критический путь (CPM)

**Функции**
- earliest start / latest start
- slack (float)
- выделение критических задач

📌 Это **переломный момент зрелости системы**

---
### 2.3 Аналитические срезы

**Функции**
- «что блокирует задачу X»
- «какие задачи можно начать сейчас»
- «что произойдёт, если сдвинуть эту задачу»
---
### Результат этапа

> Система становится **инструментом принятия решений**, а не просто редактором дат

---
## 🧑‍🤝‍🧑 ЭТАП 3. Ресурсное планирование (single-project)

### Цель

Добавить **реализм**: люди не резиновые.

---

### 3.1 Ресурсы

- человек / роль    
- календарь доступности
- capacity (часы в день)

---

### 3.2 Назначения

- task ↔ resource
- перегрузки
- конфликты
---
### 3.3 Выравнивание (Leveling)

- автоматический сдвиг задач
- сохранение логики зависимостей

📌 Это сложный этап, **не стоит начинать раньше**

---
## 🗂️ ЭТАП 4. Мультипроектность (Portfolio)

### Цель

Перейти от «проекта» к **портфелю проектов**.

---
### 4.1 Портфель

- несколько проектов
- общие ресурсы
- приоритеты
---
### 4.2 Межпроектные конфликты

- один ресурс в двух проектах
- конкуренция за сроки

---
### 4.3 Сценарное планирование

- «что если»
- сравнение сценариев
- сохранение альтернативных планов
---
### Результат этапа

> Настоящий **PPM (Project Portfolio Management)**

---
## 🤖 ЭТАП 5. AI / LLM / Assist

### Цель

Использовать ИИ **не как игрушку**, а как усилитель эксперта.

---

### Где ИИ оправдан

#### 🔹 Анализ
- выявление узких мест
- предложения по оптимизации
- объяснение причин срыва сроков

#### 🔹 Ассистент

- «что я могу начать сегодня»
- «почему проект опаздывает»
- «как сократить срок на 2 недели»
---

### Где НЕ надо ИИ

- базовые расчёты
- критический путь
- пересчёт дат

📌 ИИ — **надстройка**, не основа

---
## 🧠 ЭТАП 6. Экосистема и масштабирование

### 6.1 Управление версиями и аудит
- версии планов (снапшоты / baseline)
- аудит изменений (кто, что, когда менял)

### 6.2 Коллаборация и доступ
- многопользовательский режим
- permissions (роли: viewer / editor / admin)
- реалтайм-обновления (WebSocket / SSE)

### 6.3 Интеграции
- публичный REST API
- Webhook-нотификации
- импорт/экспорт (MS Project XML, CSV)

---
# 🔑 Ключевые принципы всей дорожной карты

1. **Сначала инварианты — потом фичи**
2. **UX-команды → домен, а не наоборот**
3. **Каждый этап даёт законченную ценность**
4. **Никакого «на вырост», только осознанные шаги**
