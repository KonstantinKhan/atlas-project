## 1. Общая оценка дорожной карты

**PO:** Дорожная карта хорошо структурирована — постепенное наращивание от фундамента к экосистеме. Мне нравится принцип «каждый этап даёт законченную ценность». Но есть проблема: мы находимся между Этапом 0 и Этапом 1, и у нас нет чёткого определения «готовности» каждого подэтапа.

**Arch:** Архитектурные решения на Этапе 0 были правильными. Разделение Task/Schedule, вынесение календаря как внешнего контекста, план как агрегат — всё это работает. Pipeline `UI → Command → API → Domain → DB → DTO → UI` реализован и доказал свою жизнеспособность.

**Lead:** Согласен по архитектуре. Но технический долг накапливается: ноль тестов, `println`-дебаг в routing, мёртвый код (`CreateTaskCommand`). Если мы начнём добавлять drag/resize без тестов, будем ловить регрессии руками.

## 2. Что реализовано хорошо

**Arch:** Несколько вещей сделаны на высоком уровне:

1. **Доменная модель** — все 5 моделей реализованы, `ProjectPlan` как агрегат с BFS-каскадом работает корректно.
2. **TimelineCalendar** — полный набор: `addWorkingDays`, `subtractWorkingDays`, `workingDaysBetween`. Инфраструктура для обратного планирования уже есть.
3. **Зависимости в домене** — поддержка всех 4 типов (FS/SS/FF/SF), детекция циклов.

**PO:** С пользовательской стороны — создание задач в пуле, переименование, назначение дат через пикеры, визуальные связи между задачами. Базовый workflow уже работает.

**Lead:** Command-паттерн на фронтенде хорошо ложится на расширение. Добавление новых команд (Move, Resize) потребует минимальных изменений в инфраструктуре.

## 3. Выявленные пробелы

### 3.1 Критические пробелы (блокируют MVP)

**PO:** Без drag/resize Gantt — это просто таблица с датами. Это ключевое UX-отличие от обычного списка задач.

**Lead:** Список критических пробелов:

| # | Пробел | Влияние |
|---|--------|---------|
| 1 | Нет drag бара | Gantt не интерактивный |
| 2 | Нет resize бара | Нельзя визуально менять длительность |
| 3 | Нет удаления задач | Невозможно исправить ошибки |
| 4 | Нет удаления зависимостей (API) | UI-удаление — фантом, пропадает при reload |
| 5 | Нет тестов | Любое изменение — риск регрессии |

### 3.2 Значимые пробелы (влияют на полноту)

| # | Пробел | Влияние |
|---|--------|---------|
| 6 | SS/FF/SF только в домене | Пользователь не может выбрать тип зависимости |
| 7 | Нет назначения из пула на таймлайн | Разделение pool/timeline неполное |
| 8 | Нет обратного планирования | Невозможно планировать от дедлайна |

### 3.3 Несоответствия карте/коду

**Arch:** Обратил внимание на расхождения:

1. **WorkCalendar vs TimelineCalendar** — в карте одно имя, в коде другое. Нужно зафиксировать `TimelineCalendar` как каноническое.
2. **Schedule команда** — в карте одна команда, в коде декомпозирована на `ChangeStartDate` / `ChangeEndDate`. Декомпозиция правильная, но карту нужно обновить.
3. **Этап 6** — слишком разнородный (версии + коллаборация + API). Нужно разбить на подэтапы.

## 4. Уточняющие вопросы

### К PO

**Arch:** Несколько вопросов для приоритизации:

1. **Drag vs Delete** — что важнее для первого релиза: drag/resize или возможность удалять задачи?
2. **Типы зависимостей** — нужны ли SS/FF/SF в MVP, или достаточно FS?
3. **Обратное планирование** — это must-have для MVP или можно отложить?
4. **Task Pool UX** — как пользователь должен назначать задачу из пула на таймлайн: drag-and-drop или кнопка «Назначить даты»?

**PO:**
1. Удаление — это базовый CRUD, без него пользователь застрянет. Но drag — это то, что делает Gantt Gantt-ом. Компромисс: **удаление первым (проще), drag вторым (важнее для UX)**.
2. Для MVP достаточно FS. SS/FF/SF — Этап 1.5 или начало Этапа 2.
3. Обратное планирование — nice-to-have для MVP. Можно отложить.
4. Для MVP — кнопка «Назначить даты» с пикером. Drag-to-assign — позже.

### К Архитектору

**Lead:** Вопросы по реализации:

1. **Drag → какая команда?** — Drag бара = `ChangeStartDate` (сдвиг всего бара, duration сохраняется)?
2. **Resize → какая команда?** — Resize правого края = `ChangeEndDate`?
3. **Оптимистичные обновления** — нужен ли optimistic UI для drag/resize?
4. **Удаление задачи с зависимостями** — каскадное удаление или запрет?

**Arch:**
1. Да, drag = `ChangeStartDate` с сохранением duration. На сервере `changeTaskStartDate` уже пересчитывает end.
2. Да, resize правого края = `ChangeEndDate`. Duration пересчитывается.
3. Обязательно. Drag без optimistic UI будет ощущаться как «залипший». Обновляем UI мгновенно, откат при ошибке.
4. Каскадное удаление зависимостей. При удалении задачи — все связанные dependency удаляются, successor-задачи остаются, но теряют ограничение.

## 5. Рекомендации

### Приоритетный порядок

**PO + Arch + Lead** (консенсус):

1. **Сначала техдолг (Фаза A)** — тесты и рефакторинг. Без этого каждый шаг — риск.
2. **Удаление задач и зависимостей (Фазы B + D1–D2)** — базовый CRUD должен быть полным.
3. **Drag + Resize (Фаза C1–C2, C6)** — ключевой UX Gantt-диаграммы.
4. **Остальное по приоритету** — назначение из пула, типы зависимостей, UX-улучшения.

### Риски

| Риск | Митигация |
|------|-----------|
| Drag/resize без тестов → регрессии | Фаза A (тесты) до Фазы C |
| BFS-каскад при drag может быть медленным | Оптимистичный UI (C6) скрывает латентность |
| Удаление задачи с зависимостями может сломать план | Каскадное удаление + пересчёт (B1 + D5) |
| Отсутствие обратного планирования ограничивает сценарии | Отложить до Фазы C5, не блокирует MVP |

### Определение «Done» для MVP (Этап 1)

**PO:** MVP считается готовым, когда пользователь может:

- [x] Создать задачу
- [x] Переименовать задачу
- [ ] Удалить задачу
- [x] Назначить даты (через пикер)
- [ ] Переместить задачу drag-ом
- [ ] Изменить длительность resize-ом
- [x] Создать FS-зависимость
- [ ] Удалить зависимость
- [x] Видеть каскадный пересчёт дат
- [x] Видеть визуальные связи

Всего: **6/10 выполнено**. До MVP — 4 задачи.
